# ADR-001: Internal account_id decoupled from Cfx identifiers

Date: 2026-06-27
Status: Accepted

## Decision

All cross-module references use an integer `account_id` generated by the database. Cfx identifiers (license2, license, fivem) are stored in an `account_identifiers` table and used only for lookup at login.

## Reasoning

Cfx identifier formats can change (license2 rollout, platform changes). Decoupling means the entire data model survives an identifier migration with a single table update rather than cascading FK changes.

## Consequences

Easier: identifier migration, multi-platform support, clean FKs.
Harder: one extra lookup at login (acceptable — happens once per session).

# ADR-002: No ox_lib dependency in v0

Date: 2026-06-27
Status: Accepted

## Decision

RPStack v0 does not depend on ox_lib.

## Reasoning

ox_lib's RedM compatibility is inconsistent and not officially documented for RDR3. Avoiding it eliminates an entire class of "works on FiveM, breaks on RedM" bugs during the most critical foundation phase.

## Consequences

Easier: zero external surprises, full control over runtime behavior.
Harder: must implement callback/RPC pattern ourselves (already done in existing rpc.lua).

# ADR-003: oxmysql + MySQL/MariaDB as sole DB driver

Date: 2026-06-27
Status: Accepted

## Decision

rpstack-persistence wraps oxmysql exclusively. No SQLite adapter.

## Reasoning

oxmysql is the de facto standard in the Cfx.re ecosystem, actively maintained, and supports async/promise patterns cleanly. SQLite adds complexity for no production benefit given RedM servers run with a DB server anyway.

## Consequences

Easier: one driver, well-documented, community support.
Harder: local dev requires a MySQL/MariaDB instance (Docker compose will be provided).

# ADR-004: Multi-character supported from v0

Date: 2026-06-27
Status: Accepted

## Decision

The identity data model supports multiple characters per account from the first implementation. `activeCharacterBySource` maps source → charId. Players choose a character after login.

## Reasoning

Retrofitting multi-character onto a single-character model requires schema and API changes that break existing resources. Building it in from the start costs little and avoids a painful migration.

## Consequences

Easier: no breaking change later, VORP/RSG parity on character selection.
Harder: slightly more complex session state (acceptable).
